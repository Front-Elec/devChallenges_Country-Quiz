# Implementation Plan: Country Quiz App

## Overview

Vite + React 18 + TypeScript SPA with React Router v6, Tailwind CSS (class-based dark mode), and the REST Countries API. The plan follows the container–presentational architecture defined in the design, building the project from the data layer upward: types → utilities → services → hooks → components → pages → routing → tests → tooling.

---

## Git Branch Strategy

Este proyecto sigue un modelo de **feature branches** sobre `main`. Cada rama representa una unidad lógica de trabajo y está asignada a un desarrollador. Ambos integrantes deben aparecer en el historial de commits del repositorio, lo cual es verificado por el docente.

### Modelo de ramas

| Rama | Dueño | Responsabilidad |
|------|-------|----------------|
| `main` | — | Solo recibe merge commits desde feature branches. Nunca se commitea directamente. |
| `feat/setup` | **Dev 1** | Scaffolding del proyecto, configuración de Tailwind, Vitest y ESLint, estructura de directorios |
| `feat/data-layer` | **Dev 1** | Tipos TypeScript, funciones utilitarias (shuffle, storage, audio), servicio de API, builder de preguntas y sus property tests |
| `feat/hooks` | **Dev 2** | Custom hooks (`useTheme`, `useTimer`, `useQuiz`), context provider y sus property tests |
| `feat/components` | **Dev 2** | Todos los componentes presentacionales |
| `feat/pages` | **Dev 1** | Componentes de página, `QuizGuard`, wiring de routing en `App.tsx` y `main.tsx` |
| `feat/tests` | **Dev 2** | Los 4 tests unitarios requeridos + lint final |

### Flujo de integración

```
feat/setup      ──merge──► main
feat/data-layer ──merge──► main   (depende de feat/setup)
feat/hooks      ──merge──► main   (depende de feat/data-layer)
feat/components ──merge──► main   (depende de feat/hooks)
feat/pages      ──merge──► main   (depende de feat/components)
feat/tests      ──merge──► main   (depende de feat/pages)
```

Cada rama se integra a `main` mediante un **merge commit** (sin fast-forward) para preservar el historial:

```bash
git checkout main
git merge --no-ff feat/setup -m "feat: project scaffolding and tooling configuration"
```

### Formato de commits (Conventional Commits)

```
<type>(<scope>): <description>

Types: feat | fix | refactor | test | chore | docs | style | ci
```

Ejemplos válidos:
```
feat(setup): initialize vite react-ts project with dependencies
feat(setup): configure tailwind css with dark mode class strategy
feat(data-layer): define shared typescript interfaces in quiz.ts
feat(data-layer): implement fisher-yates shuffle utility
test(data-layer): add property test for shuffle permutation invariant
feat(hooks): implement useTheme with localStorage persistence
feat(components): add OptionButton with feedback visual states
fix(hooks): prevent stale closure in useTimer interval ref
chore(tests): configure vitest with jsdom and jest-dom setup
```

> **Regla**: nunca incluir atribución de IA (`Co-authored-by`, `Generated by`, etc.). Ambos developers deben tener commits reales y significativos en sus ramas asignadas.

## Notes

- Tasks marked with `*` are optional and can be skipped for a faster MVP
- Each task references specific acceptance criteria for full traceability
- Sound files (`correct.mp3`, `incorrect.mp3`) must be placed manually in `src/assets/sounds/` and copied to `public/sounds/` — no automated task can fetch binary assets
- Checkpoints at tasks 4, 9, and 11 provide incremental validation gates
- Property tests (Properties 1–8) validate universal correctness guarantees and are separate from the 4 required unit tests
- The 4 required unit tests (tasks 10.1–10.4) directly map to `Requirement 10.4` and must pass for academic evaluation
- **Ambos developers deben tener commits significativos antes del merge final a `main`** — el docente verifica el historial de ambas cuentas Git
